JSDocNet

JSDocNet is an API documentation generator for JavaScript, using a comment syntax quite similar to JSDoc syntax.

JSDocNet is a C# desktop application. The user specifies a json configuration file and clicks on an execute button in order to generate the documentation. The application remembers the configuration files used in the past, so the user has just to choose one from a drop-down list.

Alternatively the configuration file can be passed as a command line argument, in which case no user interface is presented.

The goal of the JSDocNet is to be as easy as possible, both in syntax and usage, and compatible with the Microsoft Visual Studio javascript intellisense.

The configuration file

The JSDocNet generates documentation based on settings provided by a .json configuration file. The name of the file is irrelevant.

There are two ways to pass that configuration file to the JSDocNet:

  1. using the command line as \path\to\JSDocNet.exe \path\to\myconfig.json
  2. using the intuitive UI of the JSDocNet.

Here is a sample comfiguration file


    {
      "OutputFolder": ".\\DocDefault",
      "DeleteExistingOutput": true,
      "Template": "Default",
      "IncludePathList": [ ".\\" ],
      "IncludePattern": ".+\\.js?$",
      "ExcludePattern": ".+(?i)es5\\.js|.+min\\.js|.+old*.+\\.js|.+polyfill*.+\\.js",
      "DocTitle": "Default Doc Title",
      "FooterText": "Documentation Footer",
      "DisplayDate": true,
      "TutorialsFolder": ".\\DocSource\\Tutorials",
      "TutorialList": [
        "Tutorial_tp|tp Tutorial|General"
      ],
      "HomePagePath": ".\\DocSource\\Index.html"
    }

OutputFolder: Optional. The output folder where the generated documentation is placed. Defaults to .\DocDefault

DeleteExistingOutput: Optional. Delete existing output folder. Defaults to true.

Template: Optional. The name of the documentation template module to use in generating documentation files. Empty or default, means the default template module.

External documentation template modules are loaded at the application startup, scanning any .Net assembly found in the application folder, having a file name like dtm_XXXXXX.dll (dtm stands for Documentation Template Module) and gathering types implementing the IDocTemplateModule interface.

IncludePathList: Required. A list of folders or files. All these are included in parsing.

IncludePattern: Required. A regular expression. All files/folders, specified by the IncludePathList setting, that match to this setting are included in parsing. Empty means all.

ExcludePattern: Required. A regular expression. All files/folders, specified by the IncludePathList setting, that match to this setting are excluded from parsing. Empty means no excludes.

DocTitle: Required. Documentation title.

FooterText: Required. Footer text.

DisplayDate: Required. True means display the generation date in footer. Defaults to true.

TutorialsFolder: Optional. The folder where the tutorials can be found. The user may provide tutorials, as .html pages, to be included in the documentation.

TutorialList: Optional. A string list where each entry has the format TutorialName|TutorialTitle|TutorialCategory. Used to create links (anchors) for the tutorials in the generated html.

HomePagePath: Optional. The full path to an .html file to be used as content for the index.html of the generated documentation.

Tutorials

A tutorial is just an .html file placed in the TutorialsFolder as specified in the configuration file.

A tutorial is referenced in a documenation comment using the @tutorial tag

Tutorials provide more detailed information about a code feature than a DocLet. A tutorial may concern a single feature, such as a function or property, cover a whole concept, or document a step by step process.

JSDocNet accepts as tutorials just plain .html files without the html, head and body tags.

Here is the simplest, but valid, tutorial example: 


    <div>
        Here goes the text and the examples of the tutorial
    </div>

Template modules (plugins)

The JSDocNet just parses the specified .js files and produces a tree of DocItem class instances. When it comes to actual documentation generation and .html file creation it relies to a, so called, template module.

Is the template module that generates the actual documentation and the html files, leveraging the produced tree of DocItem class instances.

A template module is a class that implements the IDocTemplateModule interface.


    // Represents an object that generates the actual documentation
    public interface IDocTemplateModule
    {

        // It is passed the parser results in order to generate the actual documentation
        void Execute(DocContext Context);

        // The name for this template module, e.g. Default
        string Name { get; }
    }

The JSDocNet contains a default implementation of the IDocTemplateModule, called the default template. A user may opt to provide his own implementation of that interface, in an external module (plugin).

External documentation template modules are loaded at the application startup, scanning any .Net assembly found in the application folder, having a file name like dtm_XXXXXX.dll (dtm stands for Documentation Template Module) and gathering types implementing the IDocTemplateModule interface.

Default Template module

By default the JSDocNet uses the built-in Default Template Module when generating documentation files. The Default Template is always available.

The Default Template module uses template html files. The user may modify those template html files and especially the Layout.html file. It may also modify the .css files used by the Default Template module in order to generate a documentation of his choise.

Terminology

A documentation comment is called DocLet by this javascript documentation generator application. A DocLet, a documentation comment, may contain a number of tags.

A tag is a word prefixed by the at character e.g. @tag. Tags fall in two major categories: block tags and inline tags.

Block tags must be placed at the start of the line. Leading space and star characters are ignored. The tag place order is insignificant.

Block tags belong to one or more categories, such as doclet tags, single-line tags, multi-line tags, flag tags.

Inline tags are tags that can be placed inside text. There are just two inline tags:

  • link
  • turorial

The @tutorial tag is considered a Block tag when it is placed at the start of the line.

DocLet Tags

DocLet tags identify the whole documentation comment and the code item being documented. A documentation comment must contain just a single DocLet tag. DocLet tags may have or may have not a value. When a DocLet tag provides a value, then that value is the name of the code item being documented. Such a name may be prefixed by its memberof, e.g. tp.Classes.MyClass is parsed as name=MyClass and memberof=tp.Classes

NOTE: Name in DocLet tags is optional. When a DocLet tag does not specify a name and the @name tag is ommited, then the parser parses the next two lines after the end of the comment and tries to come up with a name.

A DocLet tag may provide the description text after the name

DocLet tags are the following:

  • namespace
  • class
  • constructor
  • constant
  • field
  • property
  • function
  • enum
  • callback
  • interface

The following DocLet tags use this syntax: @tag [MemberOf.]Name [Description]

  • namespace
  • class
  • constant
  • field
  • property
  • function
  • enum
  • callback
  • interface

The @constructor DocLet tag alone uses this syntax: @tag [MemberOf.] [Description]

The following DocLet tags may need to provide a type and so they can use this syntax too: @tag {Type} [MemberOf.]Name [Description]

  • constant
  • field
  • property
  • enum

The optional @name tag may be used in providing the name of the code item, e.g @name MyClass .

NOTE: When the @name tag is ommited and the DocLet tag does not specify a name, then the parser parses the next two lines after the end of the comment and tries to come up with a name.

Multi-line Tags

Tags that provide a description text and the @example tag, may span to multiple lines. The following are the multi-line tags:

  • namespace
  • class
  • function
  • callback
  • interface
  • enum
  • constant
  • field
  • property
  • description
  • param
  • return
  • fires
  • throws
  • example

Flag Tags

Flag tags have no value at all. They just mark the code item giving it a special meaning. Flag tags are the following:

  • function (when used inside a namespace or enum DocLet tag in order to denote the existing of a function with the same name)
  • static
  • readOnly
  • bitfield
  • eventargs
  • deprecated

ContextName

When the @memberof tag is empty, or not provided at all, then the name of the last container tag is used (called the ContextName internally).

If ContextName is empty, then the global namespace is assumed.

Setting a value to the @memberof tag, assigns the internal ContextName too.

The Fullname

The notion of the fullname applies to DocLet tags and code items. The fullname includes the full path of the item. Where the name is just the name of the item.

That is for a class named A which is a member of the tp namespace the fullname is tp.A.

Syntax

A DocLet, that is a documentation comment, uses a syntax, more or less, similar to the JSDoc syntax.


    /** This is the description
    @tag1
    @tag2
    @tagN
    */ 
    

    /**  
    @single-line-tag1
    @single-line-tag2
        When the @description tag is not provided then text lines following any single-line tag are considered the description.
    */

Here is a more realistic examle:


    /**
    @function Func
    @static
    @description this is the Func description
    @param {String} S - A string
    @param {Number} N - A number
    @param {Boolean} B - A boolean
    @param {Element|String} [ElementOrSelector=document] - Optional. A DOM element or a selector. Defaults to document.
    @return {Element} Returns an element or null.
    @see {@link https://stackoverflow.com|StackOverflow}
    @see {@link https://developer.mozilla.org/en-US/docs/Web/JavaScript|MDN}
    @tutorial Tutorial_tp
    @category Strings
    @example 
        // javascript code here
    */

NOTE: When the @description tag is not provided then text lines following any single-line tag are considered the description.

Tag Reference

@namespace

Marks a namespace. A namespace always belongs to global namespace. The memberof has no effect at all.

Syntax

    @namespace
    @namespace Name
    @namespace Description
    @namespace Name Description
Examples

    /** This is the description
    @namespace NS
    */
     

    /** @namespace NS This is the description */

A namespace may provide a function with the same name as the namespace itself. In that case the @function tag is used, in order to denote the presence of such a function. Consider the following javascript code.


    /** 
    @namespace global.tp  
    @function 
    @description This item is 3 things, a namescpace, a static class and a function  
    Here is the function
    @param {String|Element|tp.A} Selector - A selector or an element
    @return {Element}
    */        
    var tp = (function () {
        function Class(Selector) {
            // code here
        }

        var _ = Class; 
 
        /** 
        @function Func1 This is a function description
        @category Date
        */
        _.Func1 = function () { };

        return Class;
 
    })();


    // or 


    @namespace global.tp2   
    @function 
    @description This item is 3 things, a namescpace, a static class and a function  
    Here is the function
    @param {String|Element|tp.A} Selector - A selector or an element
    @return {Element}
    */   
    var tp2 = function(Selector) {
        // code here
    };

    /** 
    @function Func1 This is a function description
    @memberof tp2
    */
    tp2.Func1 = function () { };

@property

Marks a namespace, class or interface property

Syntax

    @property
    @property Name
    @property Description
    @property Name Description
    @property MemberOf.Name        
    @property MemberOf.Name Description 

    @property {Type} Name
    @property {Type} Name Description
    @property {Type} MemberOf.Name        
    @property {Type} MemberOf.Name Description
Examples

    /**
    @property {String} EventName - The event name
    @default ''
    @memberof tp.Args
    */
    

    var tp2 = {
        /** @constant {String} Version - The version of the library.  
        
            A constant or field or property can be written using the short format, as  @tag {Type} Name - Description  
            @default 2017.06.23.123
        */
        Version: '2017.06.23.123',

        /**
        @property Prop2
        @type String
        @default Tripous
        @description This is property 2  

        A constant or field or property can be written using the long format where the DocLet tag, the type tag, the name tag, and the description tag are placed in different lines
        */
        Prop2: 'Tripous',
    };
    

    var tp = (function () {
        function Class(Selector) {
            // code here
        }

        var _ = Class;  

        /** 
        @constant {String} tp.DateDelimiter - This is the DateDelimiter description. Here the short syntax is used.
        @default -  
        */
        Object.defineProperty(_, 'DateDelimiter', {
            value: '-',
            writable: false,
            enumerable: false,
            configurable: false,
        });


        /** This is the Version property description. Here the long syntax is used.
        @property tp.Version
        @type String
        @readonly
        @default 2017.06.15.1204 
        */
        Object.defineProperty(_, 'Version',   {
            get: function() { return '2017.06.15.1204 '; },
            enumerable: false,
            configurable: false,
        });

        /** 
        @property UserName
        @type String
        */
        Object.defineProperty(_, 'UserName', {
            get: function () { return _.fUserName; },
            set: function (v) { _.fUserName = v;},
            enumerable: false,
            configurable: false,
        });

        return Class;
 
    })();

@callback

Defines a callback function. A callback function does not exist in reality. It is just documented (commented) somewhere in the code just for the documentation.

Callback functions and interfaces, normally, have no associated code item. The presence of a DocLet always generates a documentation item. The actual code item is not required.

Syntax

    @callback
    @callback Name
    @callback Description
    @callback Name Description
    @callback MemberOf.Name        
    @callback MemberOf.Name Description
Examples

    /**
      @callback FilterFunc
      @param {Array} A - The array to operate on.
      @param {Object} Item - The array element to inspect.
      @return {Boolean} Returning true means that the passed object passes the filter condition.
      @memberof global
      @description Applies a filter condition to an array element and returns true if the element passes the condition.
  
      Callback functions and interfaces, normally, have no associated code item.  
      The presence of a DocLet always generates a documentation item. The actual code item is not required. 
     */

@interface

Defines an interface. An interface does not exist in reality. It is just documented (commented) somewhere in the code just for the documentation.

Interfaces and callback functions, normally, have no associated code item. The presence of a DocLet always generates a documentation item. The actual code item is not required.

Syntax

    @interface
    @interface Name
    @interface Description
    @interface Name Description
    @interface MemberOf.Name        
    @interface MemberOf.Name Description
Examples

    /**
    @interface IListener
    Represents an object that listens for notifications from a notifier
    */
 

    /**
      @function OnNote Called by the notifier when sends a note.
      @param {tp.Args} Note - The note  send by the notifier.
      @return {Boolean} Returning true means the note is handled.
      @memberof IListener 
     */
